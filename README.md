## What This Robot Does

ExampleBot is a simple delivery robot:

1. **Navigate** — drives to an object using heading control and a line-sensor stop condition
2. **Collect** — picks up the object with an arm and claw, then wall-aligns to reset heading error
3. **Deliver** — follows a line to the drop-off zone, deposits the object, clears the area

The task is intentionally simple so the *code structure* is the focus, not the game logic.

## Project Layout

```
examplebot/
├── raccoon.project.yml       # Project entry point (name, UUID, config includes)
├── config/
│   ├── connection.yml        # SSH / deploy settings
│   ├── hardware.yml          # Sensors (includes motors.yml + servos.yml)
│   ├── motors.yml            # Motor ports and encoder calibration
│   ├── servos.yml            # Servo ports and named positions
│   ├── missions.yml          # Mission list (read by raccoon-cli and BotUI)
│   └── robot.yml             # Kinematics, PID, odometry, motion constraints
└── src/
    ├── main.py               # Entry point
    ├── hardware/
    │   ├── defs.py           # Hardware definitions (normally generated from config/)
    │   └── robot.py          # Robot class (normally generated from config/)
    ├── missions/
    │   ├── m00_setup_mission.py
    │   ├── m01_navigate_to_object_mission.py
    │   ├── m02_collect_object_mission.py
    │   └── m03_deliver_object_mission.py
    └── steps/
        └── arm_steps.py      # Reusable arm sequences + defer() factory pattern
```

> In a real project, `defs.py` and `robot.py` are auto-generated by `raccoon run` from the `config/` YAML files. In this example they are written by hand so you can read them alongside the YAML and understand exactly what gets generated and why.

## Adapting to Your Robot

The config values in this example target a specific test robot. Before running on your own Wombat, adjust two things:

### Motor encoder calibration (`config/motors.yml`)

`ticks_to_rad` converts raw encoder ticks to wheel radians:

```
ticks_to_rad = (2 × π) / ticks_per_revolution
```

To measure `ticks_per_revolution`:
1. Open BotUI → Sensors (or SSH and read the encoder position directly).
2. Note the reading, then manually rotate the **wheel** (not the motor shaft) one full turn.
3. The difference is `ticks_per_revolution`.
4. Compute `ticks_to_rad = 6.2832 / ticks_per_revolution`.

For raccoonOS motors the result is typically around **1.7 × 10⁻⁵**.

> **Common mistake:** entering the raw tick count (e.g. `400000`) directly as `ticks_to_rad`
> instead of computing `2π / 400000 ≈ 1.57e-05`. With the wrong value the controller thinks
> every tick moves the wheel by hundreds of thousands of radians and the robot will not move.

### Motion constraints (`config/robot.yml`)

The `linear` / `angular` blocks under `motion_pid` set top speed and braking profile.
**These must not be zero** — a missing or zero `max_velocity` makes the robot sit still
even when commanded to move.

Run `raccoon calibrate` to measure accurate values for your drivetrain and have them
written automatically into `robot.yml`.

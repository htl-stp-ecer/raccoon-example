# raccoon-example

A clean, fully-commented reference robot built with [RaccoonOS / LibSTP](https://github.com/htl-stp-ecer/raccoon-lib).

Use this as your starting point when reading the documentation or setting up a new robot. Every file is written to be readable first — no competition pressure, no half-finished experiments, no commented-out blocks.

> 📖 **Full documentation:** [htl-stp-ecer.github.io/documentation](https://htl-stp-ecer.github.io/documentation)

---

## What This Robot Does

ExampleBot is a simple delivery robot:

1. **Navigate** — drives to an object using heading control and a line-sensor stop condition  
2. **Collect** — picks up the object with an arm and claw, then wall-aligns to reset heading error  
3. **Deliver** — follows a line to the drop-off zone, deposits the object, clears the area

The task is intentionally simple so the *code structure* is the focus, not the game logic.

## Project Layout

```
examplebot/
├── config/
│   ├── connection.yml        # SSH / deploy settings
│   ├── hardware.yml          # Sensors (includes motors.yml + servos.yml)
│   ├── motors.yml            # Motor port and calibration
│   ├── servos.yml            # Servo port and named positions
│   ├── missions.yml          # Mission list (read by Raccoon CLI and BotUI)
│   └── robot.yml             # Kinematics, PID, odometry, geometry
└── src/
    ├── main.py               # Entry point
    ├── hardware/
    │   ├── defs.py           # Hardware definitions (normally generated from config/)
    │   └── robot.py          # Robot class (normally generated from config/)
    ├── missions/
    │   ├── m00_setup_mission.py
    │   ├── m01_navigate_to_object_mission.py
    │   ├── m02_collect_object_mission.py
    │   └── m03_deliver_object_mission.py
    └── steps/
        └── arm_steps.py      # Reusable arm sequences + defer() example
```

> **Note:** In a real project, `defs.py` and `robot.py` are auto-generated by `raccoon run` from the `config/` YAML files. In this example they are written by hand so you can read them alongside the YAML and see exactly what gets generated and why.

## Concepts Demonstrated

| File | Concepts |
|------|---------|
| `config/hardware.yml` | Sensor/motor/servo declarations, `!include-merge` |
| `config/robot.yml` | Kinematics type, PID gains, axis constraints, physical dimensions |
| `hardware/defs.py` | `Motor`, `IRSensor`, `SensorGroup`, `ServoPreset` with named positions |
| `hardware/robot.py` | `DifferentialKinematics`, `FusedOdometry`, `UnifiedMotionPidConfig`, sensor positions |
| `m00_setup_mission.py` | `SetupMission`, `calibrate()`, `wait_for_button()` |
| `m01_navigate_to_object_mission.py` | `mark_heading_reference()`, `parallel()`, `.until()` with compound stop condition |
| `m02_collect_object_mission.py` | Reusable custom step, `wall_align_backward()`, `mark_heading_reference()` |
| `m03_deliver_object_mission.py` | `strafe_follow_line_single()`, combined stop conditions, parallel arm motion |
| `steps/arm_steps.py` | `seq()` composition pattern, `defer()` for runtime decisions |

## Running It

```bash
# Clone and enter the project
git clone https://github.com/htl-stp-ecer/raccoon-example.git
cd raccoon-example/examplebot

# Deploy and run on a connected Wombat
raccoon run
```

See [Getting Started](https://htl-stp-ecer.github.io/documentation/00-quick-start) if this is your first time.
